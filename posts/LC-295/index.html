🔗 🔴 295. Find Median from Data Stream

題意

中位數(Median) 是有序整數列表中的中間值。如果列表的大小是偶數，則定義中位數是兩個中間值的平均值。

  • 例如 [2,3,4][2, 3, 4][2,3,4] 的中位數是 333 。
  • 例如 [2,3][2, 3][2,3] 的中位數是 2+32=2.5\frac{2+3}{2} = 2.522+3​=2.5 。

請你實現 MedianFinder 類別：

  • MedianFinder() 初始化 MedianFinder 對象。
  • void addNum(int num) 從 資料流(Data Stream) 中添加整數 num 到數據結構中。
  • double findMedian() 返回目前所有元素的中位數。與實際答案誤差相在在 10−510^{-5}10−5 之內的答案將被接受。

約束條件

  • −105≤num≤105-10^5 \leq num \leq 10^5−105≤num≤105
  • 在調用 findMedian 之前，數據結構中至少會有一個元素。
  • 最多會調用 5×1045 \times 10^45×104 次 addNum 和 findMedian。

思路：對頂堆積

首先我們考慮最暴力的做法，即每次 addNum 後，將所有元素排序，然後計算中位數；或是在遇到 findMedian 時，將所有元素排序，然後計算中位數。但不管如何，這麼做都會導致其中一種操作的時間複雜度為 O(nlog⁡n)O(n \log n)O(nlogn)，在無法保證操作是如何分布的情況下，會導致最壞情況下的時間複雜度為 O(n2log⁡n)O(n^2 \log n)O(n2logn)，這樣的時間複雜度顯然不夠高效。而根據題目的資料範圍，我們需要找到一個插入和查詢都是 O(log⁡n)O(\log n)O(logn) 的解法。

那我們不妨從中位數的定義下手，中位數是有序整數列表中的中間值，這代表了中位數左邊會有 n2\frac{n}{2}2n​ 個元素，右邊也會有 n2\frac{n}{2}2n​ 個元素。那麼我們可以嘗試使用兩個容器維護這兩個部分，並在插入新元素時，維持兩個容器的大小關係。

雖然可以用在有序列表中二分查找插入位置，或是直接使用有序容器如 multiset 或 SortedList 的方式來維護，但通常會使用 對頂堆積 來解決。

具體來說，我們會使用兩個堆積，使其滿足以下性質：

  • hp1hp_1hp1​：最大堆積，用來存儲 ≤Median\leq {Median}≤Median 的一半數字，其堆頂是這一半數字中的最大值。
  • hp2hp_2hp2​：最小堆積，用來存儲 >Median> {Median}>Median 的一半數字，其堆頂是這一半數字中的最小值。

根據以上的性質，我們可以知道，如果兩個堆積的大小相同，中位數就是兩個堆頂元素的平均值；反之，如果兩個堆積的大小不同，中位數就是 hp1hp_1hp1​ 的堆頂元素。

而在插入新元素的時候，我們需要使兩個堆積保持前述的性質，具體步驟如下：

  1. 如果當前數量為偶數，即 hp1hp_1hp1​ 和 hp2hp_2hp2​ 的大小相同，則我們需要將新元素插入到 hp1hp_1hp1​ 中。
    • 如果新元素比 hp2hp_2hp2​ 的堆頂元素還要大，則我們需要將新元素插入到 hp2hp_2hp2​ 中，並將 hp2hp_2hp2​ 的堆頂元素移動到 hp1hp_1hp1​ 中。
    • 否則，可以直接插入到 hp1hp_1hp1​ 中，但由於新元素一定比 hp2hp_2hp2​ 的所有元素小，因此這其實也等同上述插入後再移動的操作。
  2. 如果當前數量為奇數，即 hp1hp_1hp1​ 比 hp2hp_2hp2​ 多一個元素，則我們需要將新元素插入到 hp2hp_2hp2​ 中。
    • 如果新元素比 hp1hp_1hp1​ 的堆頂元素還要小，則我們需要將新元素插入到 hp1hp_1hp1​ 中，並將 hp1hp_1hp1​ 的堆頂元素移動到 hp2hp_2hp2​ 中。
    • 否則，直接插入到 hp2hp_2hp2​ 中，但類似地，這其實也等同上述插入後再移動的操作。

此外，雖然 LeetCode 的測資範圍內不會有這種情況，但若 numnumnum 的值域和整數範圍相同，則在計算中位數時，兩個整數元素相加可能會導致超過整數範圍，因此我們可以使用 hp1.top() + (hp2.top() - hp1.top()) / 2 來計算中位數。

複雜度分析

  • 時間複雜度：O(nlog⁡n)\mathcal{O}(n \log n)O(nlogn)，其中 nnn 是輸入的數字總數。
    • 每次插入操作的時間複雜度為 O(log⁡n)\mathcal{O}(\log n)O(logn)，因為插入元素和彈出堆頂元素的操作都是 O(log⁡n)\mathcal{O}(\log n)O(logn) 時間。
    • 每次查詢中位數的時間複雜度為 O(1)\mathcal{O}(1)O(1)。
  • 空間複雜度：O(n)\mathcal{O}(n)O(n)。
    • 需要 O(n)\mathcal{O}(n)O(n) 的空間來存儲所有輸入的數字。

Follow-up

如果資料流(Data Stream)中的所有整數都在範圍 [0,100][0, 100][0,100] 內，你將如何優化你的解決方案？

由於所有整數都在範圍 [0,100][0, 100][0,100] 內，因此我們可以利用 計數排序(Counting Sort) 統計每一個數字的數量，併類似前述對頂堆的思路，將數列分成兩個部分，並使用 雙指針(Two Pointers) 維護中位數。

如果資料流(Data Stream)中的 99%99\%99% 的整數都在範圍 [0,100][0, 100][0,100] 內，你又會如何優化你的解決方案？

由於 99%99\%99% 的整數都在範圍 [0,100][0, 100][0,100] 內，因此我們還是可以利用 計數排序(Counting Sort) 統計 [0,100][0, 100][0,100] 內的數字數量。對於超出範圍的數字，則可以使用暴力檢查，或是依然用對頂堆進行處理。
